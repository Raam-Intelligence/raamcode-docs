# RaamCode Docs

Public documentation site for **RaamCode**, the Python-syntax DSL that powers Raamtrade strategies.

VitePress 1.x · Vue 3 · no framework lock-in beyond the static-site generator.

## Run locally

```bash
bun install
bun run dev          # http://localhost:5173
```

Hot reload on every file save. Editor diagnostics for broken internal links show in the terminal.

## Build

```bash
bun run build        # → .vitepress/dist/
bun run preview      # serves the built site at http://localhost:4173
```

`build` runs `bun run llms` first so `public/llms-full.txt` stays in sync with `docs/`. If you only want the LLM bundle without a full build, run `bun run llms`.

## LLM discoverability — `llms.txt`

Two files live in `public/` and ship to the site root:

- **`/llms.txt`** ([llmstxt.org](https://llmstxt.org) format) — hand-curated index. Sections for getting-started, language reference, indicator catalog, examples, concepts, error reference. One line per page with a meaningful description. Context7 picks this up automatically when registered.
- **`/llms-full.txt`** — generated by `scripts/build-llms-full.mjs`. Concatenates every page in sidebar order with `<!-- ── /docs/foo ── -->` separators so an LLM can locate the original URL. About 140 KB at last build.

The generator validates that every file on disk appears in its `ORDER` array. **When you add a new page**, append its path to `ORDER` — the script exits non-zero if it finds a page that isn't listed. (If you add it and forget to update `ORDER`, the page still gets included, just appended at the end with a warning.)

You should also add the new page to `public/llms.txt` by hand — it's curated, not generated, so the descriptions stay good.

## Content map — source of truth

Every page mirrors the actual behavior of the RaamCode language. When the language changes, the matching page changes in the same PR (docs-as-contract). Contributors with access to the main Raamtrade repository should treat that implementation as the source of truth.

If a page here disagrees with how RaamCode behaves, the behavior wins and the page is a bug. Open an issue or fix it in the same PR.

## Authoring conventions

- **Tone:** direct, technical, contract-style. No marketing fluff. Each page can be read out of order.
- **Code fences:** use `raamcode` as the language tag (aliased to `python` in the markdown renderer — `python` syntax highlighting, but the tag signals "this is RaamCode, restrictions apply").
- **Cross-links:** relative paths only (`../language/sizing`). Internal-only links resolve under `cleanUrls: true`.
- **Tables for matrices:** kwarg-per-sizing-model, error-code-per-prefix, etc. Tables scan faster than prose for reference material.
- **Every error page:** "What it means" → "Why it happens" → "How to fix it" with before/after code. Keep the shape consistent across the catalog.

## Deployment

Not wired into CI/CD yet. The build is a plain static site — any CDN works:

- **GitHub Pages** — copy `.vitepress/dist/` to `gh-pages` branch, configure `base:` in `config.ts` if not at root.
- **Cloudflare Pages / Vercel / Netlify** — build command `bun run build`, output `.vitepress/dist`.
- **Static container** — wrap `dist/` in an nginx image and deploy it like any other static site.

When deploying to a non-root path, set `base: '/raamcode/'` (or whatever) in `.vitepress/config.ts`.
